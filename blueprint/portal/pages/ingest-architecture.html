Behind the scenes

One command, directly to the DB.

The find step is only as good as the sightings behind it. Every new album runs through one resumable module — scripts/ingest-album.ts — that writes jersey sightings on every photo, so a new event is findable the moment it lands. No EXIF round-trip; the legacy three-script chain was removed in the cutover.

1

Upload

To Cloudflare Images with the album-scoped id ${albumKey}-${imageKey} — a 5409 "already exists" is an error, never an alias.

↓
2

Extract

One structured, sport-aware prompt (extractOne) — caption, play_type, quality sub-scores, and players[] for sightings. It never guesses sport.

↓
3

Embed

The caption via embedText (OpenRouter text-embedding-3-large @768) — the same seam the search query uses.

↓
4

Write — directly

UPSERT photo_metadata + photo_jersey_sightings. Never the deprecated players[] column; sport_type is set by the album trigger, not the writer.

↓
5

Findable by jersey

find_photos_by_jersey over the sightings answers "every photo of #7" — the mechanism behind "did you get that?" → "here you are."

Reprocess-in-place, idempotent, proven on BPO 2026 (119 photos, zero failures). Source: scripts/ingest-album.ts, src/lib/ai/ingest-extraction.ts, src/lib/identity/sightings.ts.
